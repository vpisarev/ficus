scoped maintenance package. Its purpose is to remove an entire class of
recurring traps (stale build caches), mechanize the bootstrap-regen ritual, and
close one leftover codegen gray area (FB-010). A successful outcome makes
several "hard-won gotchas" sections of `CLAUDE.md` obsolete — deleting them is

## WP-H2 — `tools/update_compiler.py` (do this FIRST, the other WPs use it)

Mechanize the bootstrap-regeneration recipe currently described in prose in
`CLAUDE.md` ("Regenerating the bootstrap"). Python 3, stdlib only (same
constraint as fxtest: the tooling must not depend on the compiler it manages).

Behavior:

1. Build the compiler if needed (`make -j$(nproc)`), or `--no-make` to skip.
2. `bin/ficus -O3 -o boot compiler/fx.fx` → fresh `.c` per module in
   `__fxbuild__/boot/`.
3. Diff each generated `.c` against `compiler/bootstrap/*.c`; print the list of
   changed modules.
4. Default mode: copy changed `.c` over `compiler/bootstrap/`, `make` again,
   regenerate once more and **assert fixpoint** (0 further diffs). Non-zero
   exit + loud message if the fixpoint fails (that is a determinism regression
   — point at `fxtest.py determinism`).
5. `--check`: dry run — report which modules would change, exit 1 if any
   (usable as a CI leg later; do not wire CI in this WP).
6. Clean up after itself: remove the stray `./boot` binary at repo root.

Acceptance:
- Running it twice in a row: first run may update, second run reports a clean
  fixpoint and changes nothing.
- A deliberate one-line edit in a single compiler module (e.g. a comment→code
  tweak in `compiler/Options.fx` that survives DCE) results in only that
  module's bootstrap `.c` changing.
- `CLAUDE.md`: replace the prose recipe with a one-liner pointing at the
  script (keep the fixpoint explanation, one sentence).

## WP-H1 — build-cache invalidation by compiler identity + build mode

The problem (bit us three times in Brief #2: the FB-008 mis-measurement, the
FB-011 stale `.o`, and the omp/no-omp `omp_outlined` link errors): ficus and
fxtest reuse build directories across runs, but nothing invalidates them when
the **compiler itself** or a **codegen-affecting build mode** changes. The
current workaround is folklore ("`rm -rf` the dir between measurements"),
encoded in three separate CLAUDE.md traps. Replace folklore with a stamp.

Design (keep it this simple unless something genuinely blocks):

- On (re)using a build directory (`-B <dir>` or the implicit
  `__fxbuild__/<name>/`), the compiler reads `<dir>/.fxstamp` and compares it
  to the current stamp. Mismatch or absence → treat the directory as dirty:
  wipe its cached products (or simply ignore/regenerate all of them), then
  write the new stamp. Match → current incremental behavior.
- Stamp contents, one line each, order fixed:
  1. compiler identity: md5 of the running `bin/ficus` executable
     (self-hashing via the executable path; if that proves awkward on some
     platform, an embedded build-id string regenerated by `make` is an
     acceptable substitute — document the choice);
  2. the codegen-affecting option set. Enumerate from `compiler/Options.fx`;
     at minimum: optimization level, `-no-openmp`/OpenMP mode, C vs `-c++`
     backend, `-debug`, inline threshold, and the effective
     cflags/defines that alter emitted `.c` or `.o` compatibility. Err on the
     side of including an option (worst case: an extra clean rebuild).
- fxtest: `build/fxtest/**` gets the same treatment in the harness (Python
  side) — stamp keyed on md5 of `bin/ficus`; mismatch → wipe the suite's
  cache dir before running. This specifically kills the FB-011 "stale pre-fix
  `.o` keeps failing" trap.

Explicit non-goals: no per-module dependency tracking changes, no rework of
the K-form-dump skip logic — only the outer "is this cache from the same
compiler/mode" gate.

Acceptance:
- Repro of the old trap, now green: build a program into a `-B` dir, edit a
  compiler module (the WP-H2 probe edit), `make`, rebuild the program into the
  SAME dir → the stale `.c`/`.o` are not reused (byte-compare the results
  against a clean-dir build: identical).
- Mixing `-no-openmp` and OpenMP builds in one `-B` dir no longer produces
  `omp_outlined` link errors — the mode flip invalidates the cache.
- `fxtest.py all` twice in a row: second run still uses its cache (stamp
  matches — no perf regression); after touching `bin/ficus`, caches rebuild.
- `fxtest.py determinism` green; bootstrap fixpoint via `update_compiler.py`.
- `CLAUDE.md`: delete the three traps this makes obsolete — "rm -rf the build
  dir between before/after diffs", "rm -rf build/fxtest before re-measuring",
  "don't reuse one -B dir across -no-openmp and OpenMP builds" — and replace
  with two lines describing the stamp. Also update the matching bullet in the
  "Build/run & measurement traps" list.

## WP-H3 — FB-010: 64-bit literals emitted without a suffix

Registry entry: `docs/found_bugs.md` FB-010. A folded/propagated int64
constant equal to INT64_MIN is emitted as bare `-9223372036854775808`, which C
parses as unary minus on an unsigned literal (`-Wimplicitly-unsigned-literal`)
→ wrong value/type. The typechecker's *literal* range stays as-is (symmetric,
`Ast_typecheck.fx:504`) — this WP is about C emission of constants, which can
reach INT64_MIN through folding (e.g. `-9223372036854775807 - 1`) even though
the literal cannot.

Fix: in the C-emission of integer constants, always suffix 64-bit-typed
constants — `LL` for int64, `ULL` for uint64 (blanket suffixing is simpler and
more stable than special-casing INT64_MIN; MSVC and every gcc/clang accept
it). Check both the scalar-constant path and any array-initializer /
default-value paths that print integers.

Acceptance:
- Directed unit test producing INT64_MIN via constant folding and via runtime
  arithmetic; both agree and compile warning-free.
- `tools/fxtest/cfold_gen.py`: widen the int64 generation range to the full
  `[-2^63, 2^63-1]` **for folded expression results** (keep literal operands
  within the typechecker's supported range; reach the boundary via arithmetic).
  `fxtest.py cfold` 0/500 at -O0 and -O3.
- Update the FB-010 entry to `fixed` with the usual write-up; note the
  typechecker literal-range question as explicitly out of scope (it is a
  language-design item, parked in `docs/language_changes_brief.md`).
- Bootstrap: only the module(s) you edited differ; fixpoint holds.
