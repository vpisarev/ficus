reform-prep-1 — span audit log
==============================
For each syntax construct scheduled for the reform, a minimal program that
triggers a diagnostic on or near the construct, plus the compiler's output with
the caret excerpt. This shows what source span each construct currently gets.

Legend: the "N | ... / | ^~~~" lines are the compiler's own caret rendering.
A caret under the offending TOKEN is good; the questions are (a) does it land on
the right thing, and (b) for the migrator, does the whole CONSTRUCT get a span.

NOTES (plain text, per category)
================================================================================

FOLD:
  - An error INSIDE the fold body lands on the offending sub-expression with a
    correct token span. Good for diagnostics.
  - For the imperative-fold MIGRATOR: it needs the span of the WHOLE
    'fold ... for ... {...}' form (to replace it). That whole-construct span is
    NOT what an inner error shows; it must be read from the fold node's own loc,
    which is not yet confirmed to cover the whole form (likely partial).

.op (elementwise):
  - Error on an operand lands correctly (token span).
  - For the '.op -> plain op / @ matmul' MIGRATOR: it needs the position of the
    OPERATOR TOKEN itself ('.*', '.+', ...) to rewrite just that token. The
    operator token has a span now (WP-1a), but the ExpBinary node stores the
    operator KIND, not a separate loc for the operator token -- so the exact
    '.*' position is not directly available from the node. Needs attention.

backtick:
  - Error inside the backtick lands correctly.
  - For the 'remove backtick' MIGRATOR: it needs the span of the backtick
    delimiters / the whole `...` form. Backticks are desugared during parsing,
    so the delimiter positions may be lost by the time the AST is built.

't / TYPE NAMES / TYPE EXPRESSIONS  <-- the biggest gap:
  - A type error (unknown type name) does NOT point at the type name: the caret
    lands on the preceding ':' (a single point), because typ_t carries NO source
    location. Types are structural and loc-less in the AST.
  - Consequence: the 't -> [t] reform and the type-name renames (half -> fp16,
    vector, Dynvec.t) CANNOT locate type occurrences via AST spans. A span-based
    migrator would need either (a) token-level scanning for these, or (b) adding
    source locations to typ_t (a real change).

SUMMARY:
  - Token-level spans (WP-1a) work everywhere: identifiers, operators, keywords
    all get their full width, and sub-expression errors point precisely.
  - Construct/NODE-level spans are partial for the reform targets. Ranked by
    difficulty for a span-based migrator:
      1. TYPES ('t, type names) -- loc-less, the hard one.
      2. .op operator token -- kind stored, not a token loc.
      3. backtick delimiters -- desugared early.
      4. fold whole-form extent -- needs the fold node loc confirmed/folded.
